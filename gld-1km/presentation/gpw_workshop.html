The integration problem

areal data come from many sources, none of them are tidy

19 countries · many decades · different formats

Different file formats
csv, xlsx, shp, gpkg, dbf, pdf

Different table layouts
headers in row 1, 2, or 5; multi-row headers; merged cells

Different territorial naming
“USA”, “United States”, “U.S.A.” — same place, three IDs

Different concept vocabularies
“cattle”, “bovinos”, “existencias bovinas” — same animal

Sources update
your hand-cleaned spreadsheet breaks every year

The cost of doing this by hand

Error-prone Hand-edits hide silent typos and column mis-shifts.

Unreproducible No record of what was changed and why.

Brittle Next year’s update breaks last year’s pipeline.

Two packages, one workflow

tabshiftr table reshaping

Write a schema — a formal description of where each variable lives in a raw messy table.

reorganise() mechanically applies it to produce a tidy output.

Same schema, next year’s file → same tidy output.

arealDB integration pipeline

Ingests tables and spatial geometries.

Harmonises territories against a gazetteer.

Harmonises concepts against an ontology.

Stores everything in a standardised directory.

→ tabshiftr makes the table tidy; arealDB makes the database harmonised.

1. tabshiftr

schemas for messy tables

The schema is a description, not a script

A schema describes WHERE each variable lives in the raw file. It does not describe HOW to reshape — that’s reorganise()’s job.


Raw file

year ADM1 ADM2 cattle sheep
2008 Khentii Bayan-Adarga 12450 8230
2008 Khentii Batnorov 9810 11200
2009 Khentii Bayan-Adarga 13100 8550

Schema (says where each variable lives)

setIDVar(name = "year",   columns = 1) |>
setIDVar(name = "ADM1",   columns = 2) |>
setIDVar(name = "ADM2",   columns = 3) |>
setIDVar(name = "animal", columns = c(4, 5),
                          rows    = 1) |>
setObsVar(name = "number_heads", columns = c(4, 5))

Run reorganise() → tidy long table. Same schema works on next year’s file.

The four building blocks

set* functions you compose with the pipe

setFormat() — How to interpret raw values: decimal separator, thousands separator, NA strings, flags.

setFilter() — Drop rows or columns before extraction (footnotes, totals, blanks).

setIDVar() — Identifying (qualitative) variables — what the data is about (year, region, animal).

setObsVar() — Observed (measured) variables — the numbers (heads, hectares, tonnes).

Schema patterns reference

the moves you’ll see in real tables

single column

setIDVar(name = "ADM1", columns = 2)

implicit (same for whole file)

setIDVar(name = "ADM0", value = "Mongolia")

wide ID — names spread across columns

setIDVar(name = "animal",
         columns = c(6:14), rows = 1)

obs values + unit factor

setObsVar(name = "heads",
          columns = 5, factor = 1000)

listed obs (key/value layout)

setObsVar(name = "heads",
          key = 3, value = "bovine")

format + filter quirks

setFormat(decimal = ",", na_values = c("...","-")) |>
setFilter(rows = .find(pattern = "footnote",
                       col = 1), invert = TRUE)

.find() and step-by-step debugging

.find() — dynamic cell location when positions vary across files

.find(pattern = "cattle")                          # any row, any col
.find(fun = is.numeric, col = 1)                   # only column 1
.find(pattern = "total", col = 1, relative = TRUE) # relative to cluster top-left
.find(pattern = "^total$", col = 1, invert = TRUE) # everything except matches

Debugging a schema step by step

schema_val <- validateSchema(schema = schema, input = input)   # resolve .find() calls
input_val  <- validateInput(schema = schema_val, input = input) # apply filters

getIDVars(schema = schema_val, input = input_val)      # one element per cluster
getObsVars(schema = schema_val, input = input_val)
getClusterVar(schema = schema_val, input = input_val)

reorganise(input = input, schema = schema)             # full output

schema_builder() — the easy way

an interactive Shiny app for building schemas

How it works

  1. Load your raw table: schema_builder(input)

  2. Point and click on cells in the table view:

    • mark ID variables
    • mark observed variables
    • mark clusters and filters

  3. App generates the schema and previews reorganise() output live.

  4. Copy the generated schema code into your script.

Why use it

Faster. No looking up argument names; the app exposes the right options for what you click.

Visual. You see what you’ve assigned and what’s still ambiguous before running anything.

Same output. Generates ordinary set*() calls. The schema is portable, version-controllable, scriptable.

2. arealDB

structure, harmonisation, integration

Every asset moves through three stages

vocabularies, geometries, tables — same lifecycle

stage 1 raw original

whatever the source ships — zip, xlsx, shp, pdf

→

stage 2 cleaned

UTF-8 CSV / GeoPackage; uniform format, original layout preserved

→

stage 3 normalised

harmonised against gazetteer / ontology; ready to query

The directory layout is the database

<root>/
  inventory.rds                 metadata index
  db_info.RData                 version, authors, licence, top level

  vocabularies/
    stage1/                     raw vocabulary archives
    stage2/                     UTF-8 CSV: <vName>__<dSeries>.csv
    stage3/                     <vName>_terms.parquet, <vName>_levels.parquet
    schemas/                    saved tabshiftr schemas
    mappings/                   <vName>_<dSeries>.csv: source_label → canonical_id

  geometries/
    stage1/                     raw geometry archives (zip, gpkg, shp, ...)
    stage2/                     GeoPackages, named by arealDB convention
    stage3/                     one gpkg per top-level unit (per nation)

  tables/
    stage1/<dSeries>/           raw downloaded files, grouped by source
    stage2/                     UTF-8 CSV copies, named by arealDB convention
    stage3/                     one .rds per top-level unit, harmonised
    schemas/                    saved tabshiftr schemas

Two functions per asset: register and normalise

asset register normalise
vocabulary regVocabulary() normVocabulary()
geometry regGeometry() normGeometry()
table regTable() normTable()


regX()

Records metadata in inventory.rds. Waits for you to place the actual file in stage1 / stage2.

normX()

Reads stage2, harmonises against gazetteer / ontology, writes stage3. Vocabularies are normalised first.

Vocabularies: backbone vs. mapping

the first contributor sets the canonical IDs; everyone else maps to them

Backbone

First dataseries registered against a vocabulary.

Defines canonical IDs and class structure.

normVocabulary() writes: <vName>_terms.parquet <vName>_levels.parquet

Schema can be term-grain (one row per term) or leaf-grain (one row per leaf, ancestors as columns).

Mapping

Every subsequent contributor.

Source labels are matched against the backbone.

Confirmed mappings persist in: vocabularies/mappings/ <vName>_<dSeries>.csv

Unmatched labels open match_builder() Shiny app for interactive resolution.

Geometries: spatial boundaries

one per administrative level

regGeometry(...,                # named territory selector, e.g. ADM0 = "Mongolia"
            gSeries,            # dataseries name (e.g. "gadm")
            label,              # mapping gazetteer class -> column name
                                # list(ADM1 = "NAME_1")
            archive,            # stage1 filename
                                # "archive.zip|inner.gpkg" for compressed sources
            [...],
            overwrite = FALSE)


Place the file Raw archive → geometries/stage1/ GeoPackage → geometries/stage2/<filename>

CRS must be EPSG:4326 normGeometry() harmonises every stage2 file against the gazetteer; writes one stage3 GeoPackage per top-level unit. Unmatched names open match_builder().

Tables: where the schema meets arealDB

regTable(...,                 # named territory selector, e.g. ADM0 = "Mongolia"
         subset = NULL,       # optional label, no underscores; appears in stage2 name
         dSeries,             # dataseries name for the table
         gSeries,             # dataseries name for the geometry to join against
         label,               # gazetteer level: "ADM0", "ADM1", "ADM2"
         begin, end,          # integer years
         schema,              # tabshiftr schema (must exist in global env)
         archive,
         archiveLink,
         downloadDate,
         updateFrequency,
         diagnose = FALSE,    # test-run reorganise() without saving
         overwrite = FALSE)

normTable(pattern   = NULL,
          query     = NULL,        # optional dplyr-style filter
          ontoMatch = NULL)        # column(s) to match against an ontology, e.g. "animal"

Vocabularies first — geometries and tables match against them.

Every country file follows the same pattern

  1. dataseries · 2. geometries · 3. tables

1

regDataseries()

one call per source — geometry provider, statistics agency, etc.

2

regGeometry() ×N + normGeometry()

register one geometry per administrative level (ADM0, ADM1, ADM2), then normalise all of them at once

3

regTable() ×N + normTable()

register one table per subset (e.g. one per animal); ontoMatch resolves the concept column against the ontology

End-to-end checklist

Adding a new data source

  1. Find the source, locate the data
  2. Place raw files in stage1/ subfolders
  3. Inspect raw tables with View(read_csv(..., col_names = FALSE))
  4. Write the tabshiftr schema; test with reorganise()
  5. regDataseries() for every source
  6. regVocabulary() if new terms; then normVocabulary()
  7. regGeometry() — export GeoPackage (EPSG:4326) to stage2/
  8. normGeometry() — resolve unmatched names in match_builder
  9. regTable() with diagnose = TRUE first
  10. normTable() — resolve concepts in match_builder
  11. Verify: readRDS(stage3/<TopUnit>.rds)

Common failure modes

Watch out for

regGeometry / regTable errors regDataseries() not called first

Function hangs at readline prompt file not yet in stage1 / stage2 — place it, press a key

adb_path not set new R session; re-run adb_init()

reorganise() returns wrong columns schema column numbers don’t match col_names = FALSE view

NAs where values expected na_values in setFormat() doesn’t cover actual NA strings

Unmatched territories label points to wrong column, or CRS isn’t EPSG:4326

Schema works in dev but normTable fails stage2 CSV re-saved by Excel — re-export UTF-8
